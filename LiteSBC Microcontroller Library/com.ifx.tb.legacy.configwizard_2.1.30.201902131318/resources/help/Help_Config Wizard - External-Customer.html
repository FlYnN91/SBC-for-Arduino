Introduction into Config Wizard

Overview over GUI Elements

Config Wizard is a GUI-based configuration tool that has the following main characteristics:

  • Offline mode to configure microchip registers (timers, interrupts, I/O,…) to include into existing source code, e.g. for the Embedded Power ICs
  • Online mode to communicate with certain microchips via Infineon uIO directly, e.g. for SBC and Bridges
  • Integration into Keil µVision 5
  • Many visualization elements to display current values
  • Code generation for initialization
  • Free configurable GUI for new controllers and special use cases
  • Configurations can be saved and loaded


When Config Wizard is started it normally shows an overview of available microcontrollers from which one can be selected by mouseclick. This picture shows different chip families in three tabs ("System Basis Chips", "Embedded Power ICs”, "Bridge Driver"). Depending on your version/delivery you may have more or less controllers in the selection or possibly a fixed one is loaded directly after startup.

Overview over Menus

The Config Wizard has many additional functions that can be performed via the menus.

The "File" Menu

In the file menu there are functions to load and save the current state of the UI.


Most of the UI elements shown in Config Wizard have a state that can be saved to and loaded from a so called icwp file. This mainly makes sense for offline UIs, because the elements there represent certain device properties. So when a controller is set to a defined state by the user this state can be saved to a file. In online mode this only makes sense for the Table element of the Register View, because all other elements are changed for each new command sent to the device. The Table has its own save button.

  • Home: goes back to the main selection of microcontrollers.
  • Load: loads an icwp file and sets the UI elements according to its content.
  • Save: saves the current state of the UI to a file.
  • Save As: saves the current state of the UI to a file, allowing the user to change the name.
  • Exit: leaves the program. If there are unsaved changes (state changes of the UI elements) the user is asked for saving it.
  • Export Header Files: writes the register values represented by the UI elements to one ore more 'C' header files

The "Extra" Menu

This menu contains many different functions:

  • Open XML: opens an xml file containing the definition of a UI (not to be confused with an icwp file that contains the state of such an UI). This is rarely used as normally such a file is loaded as described in overview over GUI elements with clicking on the symbol of a controller after startup.
  • Reopen XML: reloads the current loaded UI and sets back all UI elements to their initial values.
  • Search GUI element: searches for a certain UI element and thus in normally for a register. It opens a dialog containing a combobox where the user can (supported by a completer list) enter a register name. This is mainly used in the offline mode and therefore not applicable for SBCs and Bridges.



The Config Wizard opens and selects the UI element associated with the selected register. If the element is currently not enabled it can't be selected and flashes in red for a short time along with its tooltip opened.

  • Show Differences: lets the user compare the values entered in the current UI with the values saved in an icwp file before. After selecting an icwp file to compare with, a dialog showing the differences appears:

    Next to the UI / register name the current and the file value are shown for different values. When the user clicks on the register name, he is navigated to the associated UI element.
  • Expand branch: expands all trees (or a selected one) on the screen.
  • Collapse branch: closes all trees (or a selected one) on the screen.
  • Settings: opens a dialog where some system settings can be altered:


The "Scripts Directory" is the start directory where the UI xmls are searched (not to be confused with the files containing script commands controlling the UI).

When the combo box is opened a least recent directory list is shown.

The "Working Directory" is the directory where the icwp files are loaded and saved and where the header files are stored. This directory can't be set by the user but is set by the program dynamically e.g. when an icwp file is loaded or stored.
"uIO  Logging" makes sense in online UIs only where the data traffic to and from uIO can be logged. A typical log output into the console window would look like this:



The green lines are the data to uIO, the blue ones from uIO. Logging is mainly needed for developing an UI xml and slows down execution of the Config Wizard and reaction of the UI, so it only should be used when necessary.

The "Scaling factor" is in experimental stage currently and maybe not enabled in your version. It is used mainly for scaling on 4K monitors.

  • Get uIO Version: gets the firmware version of a connected uIO device and (if available) compares with the version of the currently loaded XML. The firmware of the respective uIO has to be updated manually. It then shows a messagebox with the firmware installed on the uIO and shows if this version is sufficient to run the current UI XML.
  • Update uIO: lets the user update the uIO to another firmware version. After a warning, if uIO is connected already to the UI XML it asks the user for a "hex" file containing the firmware for the uIO device, usually contained in a subfolder "uIO-Stick_Firmware". The update itself takes about a minute. Don't remove the uIO device during update process!
  • Run script file: runs a script file to automate operation.
  • Record script: Records the user actions on GUI into a script file.
  • Add script command: Adds a special command to the recording script.
  • Sign Configuration File: calculates a checksum over a UI xml file and inserts this checksum into the file. This is done to ensure that such a file is not changed by accident, resulting in unpredictable behavior and/or wrong register values. If a message like "File xxx.xml has been manipulated. Do not use it!" appears, the file is not the original one delivered by Infineon. You should only sign a file when you are sure what you are doing! This menu entry may be missing in your version of Config Wizard.
  • Validate XML: checks a UI xml file by many criteria for correctness. This is only useful for XML developers only, as UI XMLs distributed by Infineon are tested and should not contain errors. This menu entry may be missing in your version of Config Wizard.
  • Show locks: let the developer of an XML check if all lockings are correct.This menu entry may be missing in your version of Config Wizard, because it's useful for developers only.
  • Full screen: switches to full screen mode. Full screen mode can be exited with Ctrl-Shift-F.

2 Command line options

The Config Wizard can be started with several command line options that are enumerated in the following:

ConfigWizard [<Filename.xml|icwp>] [-g] [-q] [-s] [-b<Batchfile>] [-c<Outfile>] [-o<Outfile>] [-?]

  • -g                          Generate Header Files (non-interactive)
  • -q                          Quiet mode (no popup massages on errors)
  • -s                          Save Project File, with correct HASH, and generate Header Files
  • -x                          Terminate program after Keil integration
  • -k                          Enable Keil µVision Integration
  • -b<Batchfile>        Run batch file
  • -c<Outfile>           Set console logging file (for output of console and, if in quiet mode for message box output)
  • -o<Outfile>           Set output logging file (for output of batchfile/scripting)
  • -?                          Help

Main Use Cases 

There exist online and offline type of the UI. Both types are not excluding each other mutually by design, but usually they do not overlap.

Offline mode, e.g. for Embedded Power ICs

The GUI described below is a typical representative for an offline UI, that means that the user can change the settings of the microcontroller and export these settings into a 'C' headerfile. This headerfile typically is included in a user supplied software and the #defines in the headerfiles are written into registers to initialize the controller on startup of the software.

After selecting a controller in the start screen, the UI for the selected chip is loaded. A typical user interface could look like this (describing the SCU (=System Control Unit) part of the TLE987 device):


 

At the top there is a menu, which has the following functions:

  • "Home" (Ctrl + N) which goes back to the start screen allowing to select another device. If changes were made you are asked to save the current state in a so called "icwp" file.
  • "Load" which allows you to load such an icwp file.
  • "Save" (Ctrl + N) saves an icwp file
  • "Expand" expands all trees (or a selected one) on the screen
  • "Collapse" closes all trees (or a selected one) on the screen

The "Tab" element is an opportunity to group related functional units together (and to omit overloading the screen). In each tab only the elements of the device that concern a certain functionality are shown.

The central screen is populated with several elements like

  • Tree: groups elements in a hierarchical manner
  • Checkbox: is typically used for binary selection (enable/disable, on/off,…)
  • Radiobuttons: are used to select from a list of mutually exclusive alternatives
  • Combobox: also let the user select between options
  • Spinbox, which allows to enter numerical values. Also available in a hexadecimal and a floating-point variant.
  • Calculations: combine the values of several elements and writes a result to screen. In the example above a frequency is calculated from e.g. a base frequency, divider, etc.
  • The statusline finally shows the currently loaded GUI xml along with its version.

Checkboxes and radiobuttons not only can let the user select different values but also can have so-called "branches", sub-elements that are only accessible when the head of the branch is selected. Examples are the "Override Internal Oscillator Frequency" checkbox that enables the “Int. fOSC” spinbox or the marked Radiobuttons that enable either "Automatic Clock Configuration" with its "fSYS" combobox or the "Manual Clock Configuration" with a large subtree of elements.

Other elements not present in the picture above are edit fields where a text (e.g. the name of a callback function) can be entered, or buttons that mainly are used to send data to a device, LEDs to indicate a status, images or a set of elements  that can show and log data incoming from a uIO.

As one can see, not all elements are usable, some are disabled. This is another feature, called "locking". Certain UI elements disable (in the sense of the user can't change them) other elements (and set them to a predefined value), because the controller function determined by the first UI element forces another element (that represent another controller function) to be set to a fixed value. A typical example would be: the user enables an UART. This UART uses a certain pin for its TX wire. But this wire could also be configured as a GPIO output from another UI element. So, if the UART is enabled, the associated GPIO UI element would be disabled.

When the user hovers over the UI elements with the mouse, over most of them a tooltip appears that shows the most importants attributes of the UI. You can see the associated register (or sub-part of a register), the header file into the value would be exported and the UI elements that this element is locking (or, if it is locked itself, the UI element which locks it, would be shown):



Online mode, e.g. for SBC or Bridges

Another type of the UI would be the online type (both types are not excluding each other mutally by design, but usually they do not overlap). The online type communicates via a uIO module (which converts USB to LIN / SPI) with a controller that in turn has interpreter software running on it. To this interpreter commands can be written to configure / start functions of the controller. Please note that the uIO must be plugged in before the XML file is loaded.

A typical UI for the online type looks like this (status line, etc. omitted):

Essential newly added elements are the LED that normaly shows a binary state (here e.g. the states of the uIO and the connected 9263 chip) and the pushbutton that is used in this example to send commands to the controller via the uIO. The commands itself are built up from several UI elements, and-ing and or-ing bits together. As we can see another possibility to build the commands is given starting the Config Wizard with the register view.

The upmost group "Register Selection" shows information taken from a svd file (containing information about the registers of a controller). The first combobox lists all subsystems of the chip, in the svd designated as peripherals.

These peripherals contain registers, which in turn are made up of fields, one or more bits responsible for a certain function.

The user selects a peripheral ("Register Domain Selection") and all registers contained in this peripheral are copied into the second combobox ("Register Selection"). The fields contained in the register there selected are shown in the table below, the registerview. Often there is a description available in the svd file, this is shown in tooltips in the several elements.

Now in the registerview for each field the desired value can be selected. In most cases there are predefined values for the field-values, so called "enumerates". If enumerates are available, a combobox opens when a field is selected, containing the values in a textual form:

When there are no enumerates in the svd then the field can be edited (also in place)

  • in 1 bit wide fields (bool) with a checkbox
  • in >1 bit wide fields with a spinbox (range is derived from width)

The header line besides the field name contains the bits affected and the read/write or read-only characteristic.

Right above the registerview there is a radiogroup that can be selected to either "Read" or "Write/Clear". This special chip needs this information too (other maybe won't), it will be combined with the address when sent.

When the register is filled as requested, the button "-->" is pressed and the register and its value is copied to the next line of the table (or the current selected line is overwritten, depending on which edit mode is selected).

The table can be edited in the following manner:

  • Lines can be deleted
  • Lines can be moved via Drag-And-Drop
  • Some columns can be edited directly (W/R, Value, Delay and Comment).
  • Lines can be copied with Ctrl-C, Ctrl-V.
  • The contents of the table can be saved in a file and loaded from there.

Numerical values can be entered as decimal, hexadecimal (with a "0x" prefix) or binary (with a "0b" prefix).

The "Value" column has a tooltip showing what the value means, in deciphering the value into enumerates.

The table can be sent to uIO (starting from the selected line), where after a line a delay (in msec) can take place, or a single step mode can be selected (after each sent command a messagebox asks for continuation or stop). The answer from the uIO device is written to the "Answer" column.

Also the content of the table can be exported as a "C" function into a file, so the sequence can be added to own code.
